# 加速 Rust 编译：第一篇 —— 等待的艺术

*三篇系列之一。[第二篇：缝隙](blog-part2-zh.md) | [第三篇：填缝](blog-part3-zh.md)*

## 你的 17 分钟，就这么没了

我们来聊聊 Zed。

Zed 是一个用 Rust 写的代码编辑器。快。漂亮。现代。那种让你觉得"Rust 真香"的项目。

然后你试着从源码编译它：

```
$ time cargo build --release
...
Finished `release` profile in 16m 52s
```

十七分钟。

你启动编译。你去看邮件。你去倒咖啡。你喝完咖啡。你刷了会儿手机。你开始怀疑人生。编译完了。你发现有个 typo。你重新来。

这不是 Zed 的问题。这是 Rust 的问题。或者更准确地说，是*大型 Rust 项目*的问题。Zed 有超过 50 万行代码，分布在 198 个 workspace crate 中。编译器要把这些全嚼一遍。

但我们肯定能搞得更快吧？Rust 社区优化编译器都优化了这么多年了。让我们把能试的都试一遍。

## 第一招：并行前端

Rust nightly 有并行前端。多线程嘛，一定更快。

```bash
RUSTFLAGS="-Z threads=8" cargo +nightly build --release
```

结果：快了大概 5-10%。不错，但还是要等 15 分钟。并行前端帮的主要是解析和类型检查，但真正吃时间的是 LLVM 代码生成——那本来就是按 codegen unit 并行的。

## 第二招：换个更快的链接器

链接需要时间。我们用 `wild`，一个用 Rust 写的快速链接器：

```bash
RUSTFLAGS="-C link-arg=-fuse-ld=wild" cargo +nightly build --release
```

结果：链接时间从 8 秒降到 3 秒。对链接来说很棒。但链接只占总编译时间的不到 1%。我们从 1,012 秒里省了 5 秒。瓶颈不在链接。

## 第三招：编译缓存

`sccache` 可以缓存编译结果，这样下次编译就快了：

```bash
RUSTC_WRAPPER=sccache cargo build --release
```

结果：*第二次*编译确实快得飞起。但*第一次*编译——全新的、干净的编译——跟没缓存一模一样。而在 CI 里，每次都是全新编译。你的新同事第一次 `git clone && cargo build`？全新编译。切换到一个依赖不兼容的分支？全新编译。

缓存并没有减少工作量。它只是记住了上次做过的活儿。

## 第四招：Cranelift 后端

不用 LLVM 了怎么样？Cranelift 后端编译快得多：

```bash
RUSTFLAGS="-Z codegen-backend=cranelift" cargo +nightly build
```

结果：编译确实快了不少。但生成的代码没有优化。Cranelift 适合开发构建，但做 release 构建——就是给用户用的、CI 产出的那种——你需要 LLVM 的优化。我们需要的是让 `--release` 变快。

## 第五招：优化 rustc 本身

Rust 项目已经用 PGO（基于性能分析的优化）来编译发布编译器了。多年的努力已经让 rustc 本身尽可能地快。你正在用的 nightly 已经享受到了所有这些优化。

然而，十七分钟。

## 一个灵魂拷问

让我问你一个有点不舒服的问题。

我们试了并行前端。试了快速链接器。试了缓存。试了替代后端。试了优化编译器本身。

**如果编译器已经做得够好了呢？如果问题不在于编译器*怎么*编译，而在于我们让它编译了*什么*？**

想想看。当你对 Zed 执行 `cargo build --release` 的时候，编译器老老实实地编译每个 library crate 里的每一个公开函数。`regex` crate 暴露了几十个函数——你的项目可能只用了三个。`serde` crate 有几百个方法——你用了其中一小部分。编译器不知道这些。它也不可能知道。它在独立地编译每个 crate，而任何公开函数都*可能*被下游调用。

如果编译器做的工作里，有一大块其实是……不必要的呢？

如果我们能在编译开始之前就告诉编译器，"嘿，这 9,000 个函数你不用管了"？

这就有意思了。
